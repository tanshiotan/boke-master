# 🤖 CLAUDE.md - プロジェクト開発・運用ガイドライン

## 1. 役割分担と基本運用
- **メインモデル（司令塔・編集長）**: 全体設計の統括、不変条件の厳格な監査、およびPRタイトル・本文の「誰が読んでも分かりやすい丁寧な日本語」への推敲・執筆を担当します。
- **サブエージェント（Sonnet / 実務部隊）**: コードベースの静的解析、ソースコードの修正、Gitブランチ操作、プッシュ、品質検証（テスト・フォーマット）の実務を担当します。

---

## 🛠️ 2. 運用・Git作業規約

1. **原則ルール**: 「1プロンプト・1ブランチ・1機能」を死守してください。
2. **ブランチの起点と命名規則**:
   - 必ずベースブランチ（指定がない場合は `main`）を最新状態にしてから起点としてください。
   - 適切なプレフィックス（`feat/`, `fix/`, `perf/`, `refactor/`）を用いた具体的な名前を生成してください（例: `fix/finalize-loading-flicker`）。※単なる連番（`pr1` など）は禁止。
3. **プッシュ前の品質チェック**:
   - 修正後、対象ディレクトリにおいてコードの自動整形と静的解析・型チェック（frontend: `npm run fmt` / `npm run lint`、backend: `make fmt` / `make lint`）を必ず実行し、エラーがないことを確認してからリモートへ push してください。
4. **PRのマージ先（Base）指定**:
   - PRを作成する際のマージ先は、指定された適切なベースブランチ（`develop`, `main`等）を指定してください。

## 💻 3. コーディング・実装規約（堅牢性担保）

実装にあたり、以下のコード品質規約を**絶対厳守**してください。

### 共通・言語規約
1. **不変性の担保 (TS/JS対象時)**:
   - 予期せぬ再代入によるバグを防ぐため、`let` の使用は最小限に抑え、原則 `const` を使用した堅牢性の高いコードにすること。
2. **Python型ヒントのモダン化と明示化 (Python対象時)**:
   - 古いバージョンの大文字型ヒント（`List`, `Dict` 等）は使用せず、小文字の `list`, `dict` に統一すること。
   - `list[tuple[Product, int, int]]` のような要素の意味が不透明で可読性の低い多重構造の型定義は禁止する。必ず `BaseModel` (Pydantic) や `Dataclass` で明示的に型を定義すること。
3. **レイヤーの分離 (Router層の責務)**:
   - `/router`（エンドポイント定義）内で直接データを操作・加工しないこと。データ操作やビジネスロジックはすべて `Service層` に記述すること。
4. **型安全なデータ操作 (実行時エラーの防止)**:
   - 配列のインデックス直接指定や、ディクショナリの文字列キー直接指定による生データ操作は禁止する。必ず `BaseModel` や `Dataclass` に変換し、ドットアクセスで操作すること。
5. **マジックナンバーの排除**:
   - 実数や固定文字列をコード内に直接記述（ハードコード）せず、必ず名前つき定数として定義すること。
6. **共通化リファクタリングのプロセス**:
   - リファクタリング時は機能追加とはコミットを分け、動作確認完了後に別コミットとして共通化処理を入れること。
7. **コード内コメントのルール**:
   - コメントには**句読点（「、」「。」「,」「.」）を一切入れない**こと。意図が最短で伝わるよう簡潔に記述すること。

### フロントエンド・データ取得・UI規約
8. **データ取得（TanStack Query）**:
   - 非同期データの取得には原則 **`useSuspenseQuery`** を使用すること。
   - 個別のコンポーネント内で `isLoading` や `isError` による分岐は記述せず、必ず **`SuspenseQueryBoundary`**（Suspense + ErrorBoundary統合コンポーネント）で囲んで宣言的にハンドリングすること。
9. **UIコンポーネント・フィードバック**:
   - アイコン表示にはプロジェクト共通の **`Icon`** コンポーネント（`react-icons` 抽象化）を使用すること。
   - ユーザーへの通知・フィードバックには **`sonner`**（トーストヘルパー）を使用すること。
   - 削除や重要なアクションの確認には共通の **`AlertDialog`** コンポーネントを使用すること（`window.confirm()` や `alert()` 等のブラウザ標準APIは使用禁止）。

## 🔍 4. コードレビュー規約

### 🚨 絶対ルール
1. **コードの修正（コードブロックでの書き換え）は絶対にしないでください。**
2. 指摘や提案は、修正箇所の特定が容易なようにファイル名・行数・具体的なコードを明記し、文章で簡潔に出力してください。
3. スコープ外の無関係な指摘（好みの問題、今回のテーマとは異なるリファクタなど）は避けてください。

### 📋 レビューの3大観点
1. **修正の妥当性（真っ当な修正か）**:
   - PRの目的を達成するために、実装アプローチが最適かつシンプルか。
   - `useSuspenseQuery` と `SuspenseQueryBoundary` の組み合わせなど、アーキテクチャに準拠しているか。
2. **対応の取りこぼし（修正漏れ）**:
   - 同様の対応が必要なはずなのに修正から漏れているファイルや箇所がないか。
3. **開発規約ルールの準拠確認**:
   - `npm run fmt` / `npm run lint`（backendは `make fmt` / `make lint`）が通るコードか、上記「コーディング・実装規約」に違反している箇所がないか。
